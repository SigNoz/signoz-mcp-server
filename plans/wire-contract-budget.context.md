### 2026-07-19 — Description-quality research and second-pass review
- Reopened the completed wording pass after the user asked how description content should be chosen. Reviewed the official MCP schema/tools guidance, OpenAI function-calling and Apps SDK metadata guidance, and Anthropic tool-definition guidance before making further edits.
- The 1,024-byte ceiling is a project delivery budget, not a universal MCP/OpenAI/Anthropic rule and not a target length. Optimize for correct selection and invocation within the ceiling rather than for the shortest possible label.
- Use the tool description as a selection contract: say what the tool does or returns, when to choose it, when a nearby tool is the better choice, and any non-obvious prerequisite or invariant that the model must know before calling. Put field meaning, format, units, defaults, ranges, enums, and field-local examples on the parameter schema; put result shape in `outputSchema`; put read/write/destructive/idempotent semantics in annotations; put shared workflows in server instructions; put long grammars, field catalogs, and complete examples in resources; put actual-call repair advice in execution errors.
- OpenAI recommends one or two discovery-focused sentences beginning with “Use this when…”, while Anthropic recommends enough detail (often three or four sentences for complex tools) to make relationships and caveats explicit. For this multi-client MCP server, use three compact, high-signal sentences by default and a fourth only when a critical call rule cannot safely move elsewhere.
- `signoz_list_alerts`: the old description had a strong instances-vs-rules/history boundary but duplicated every filter example, pagination default, and output field. The first trim preserved nearly all facts and therefore remained parameter-like. Keep the boundary and result summary; leave syntax/defaults to existing property descriptions and the pagination output schema.
- `signoz_create_alert`: the old description mixed selection, the complete schema split, resource contents, notification recovery, supported types, and label rules. The first trim retained the important notification workflow and resource route, but did not distinguish creation from update. Add that boundary; keep the channel-selection prerequisite because it affects the first call, while relying on the typed schema, alert resources, validation errors, and channel-selection error for payload details.
- `signoz_create_dashboard`: the old description was mostly a resource index plus a repeated required-field list. The first trim safely moved the query-specific index into the widget guide, but still lacked the important custom-dashboard-vs-curated-template boundary. Add `signoz_import_dashboard` as the negative route and keep only the core resource pointer in the tool description.
- `signoz_update_dashboard`: the first trim improved the old prose by adding fetch/merge/preserve guidance. Make selection explicit and state that the operation is a full replacement, not a patch; retain the core resource pointer for changed widgets.
- `signoz_execute_builder_query`: the first trim preserved correctness but stayed close to the ceiling because selection, routing, argument construction, normalization, PromQL syntax, and an external docs URL all remained top-level. Keep the raw escape-hatch boundary and resource routing in the tool description; move `spec.limit`/`spec.order` and 100/10000 normalization to the sole `query` property's description, where argument construction belongs.
- Logs/traces `requestType`: the old prose repeated many examples and agent workflow rules. The first trim kept “ask if ambiguous” and “make one call”, but those are not field semantics and the server already owns the no-redundant-query rule. Keep only the default and the semantic choice between a whole-range scalar/grouped result and per-bucket time series.

## Open Questions
- [x] Should the four wide schemas be redesigned here? No. Issue #158 explicitly limits this change to exact inventory pinning; a typed-object redesign would be breaking.
- [x] Should source or manifest text be used for the budget assertions? No. The acceptance criteria require initialize/`tools/list` wire output.
- [x] Does this require live SigNoz verification? No. No upstream request/response behavior changes; protocol-level in-process verification is the relevant boundary.
- [x] Should the guard preserve the issue's incomplete four-tool/four-schema inventory? No. Enforce the stated catalog-wide limits against current wire reality and document the two discrepancies.
- [x] Should this change add total-schema-byte or nesting-depth ceilings? No. Their thresholds remain undefined in #34/#158, so they cannot be pinned here without inventing policy.
- [x] Is passing the byte guard enough to judge description quality? No. The descriptions also need explicit positive/negative selection boundaries and critical invocation guidance, checked statically here and ultimately validated with direct, indirect, and negative prompt evaluations.
